This is one round of an ongoing review, not a one-shot. We run it again and again until a round turns up nothing real, so this might be the first round (feature just built, I've done my manual testing) or the Nth (we've already found and fixed bugs in earlier rounds). Doesn't matter, the job is identical every time: find everything wrong with the current uncommitted code and fix it cleanly. So please don't burn time re-establishing "where are we, is this fresh." Just read the current full diff (it already includes whatever got fixed in prior rounds), review all of it, fix what's real. The point is to keep the round count low, cast a wide net each time so we don't get nasty surprises and have to run ten more.

Start by exploring, take as long as you need. Read the full `git diff` of everything that's going to be committed. Most of this code is NOT in your context, so do not review from memory. You'll naturally over-focus on whatever you touched most recently and forget the rest, actively fight that, the mandate is to cover the entire uncommitted change, every file, not just your latest fixes. You can bias one or two agents toward the riskiest or most recent work (including earlier-round fixes, which is exactly where new bugs hide), but overall coverage of the whole diff is the priority. Prior-round fixes are just part of the diff now, they get reviewed like everything else, no "already handled" status. Use the diff to carve the work into well-scoped agents that together cover everything with no gaps.

Then launch all the review agents in parallel, in a SINGLE message with multiple Agent tool calls, all of them the `code-reviewer` agent type. Never one at a time, never sequentially waiting for one before the next, this is a hard rule: do all your exploration first, decide the scoping for every agent, then fire them all at once in one batch. Sequential launches waste enormous time and are unacceptable. Cast as wide a net as you reasonably can, more well-scoped agents beats too few. For each agent, pass along the global rules relevant to what they're reviewing, they don't share your CLAUDE.md or MEMORY.md, so transmit the project conventions and standards they need to judge the code correctly.

The review agents run on Fable and they are sharp at finding architectural holes. Each one returns not just findings but a proposed fix per finding. That proposal is an input to YOUR decision, not the final word: the agents see only a slice of the rules and a slice of the architecture, you hold the whole picture. So your job on every proposal is to judge it against the full architecture and the complete project rules, then accept it, sharpen it, or replace it with something better. Expect to override some of them. When an agent flags a genuine fork (two defensible fixes turning on a tradeoff it couldn't resolve), that is the one case it hands you a real choice, decide it from the rules plus context, or surface it to me only if it's a true cornelian fork you can't resolve either.

As results come in, validate each finding before you trust it, and validate it on TWO axes, not one. First: is the issue real, or is it a deliberate choice the agent lacked context for. Second, and this is the one I keep getting burned on: is the agent's PROPOSED FIX the right shape, or is it a correct-but-dirty local fix that duplicates something the architecture already provides. The failure mode: an agent flags a real issue and proposes a direct local fix (a macro, a hand-rolled validation, a side path, user-facing boilerplate) when the codebase ALREADY exposes the right mechanism and the real fix is just to connect to it. Concrete past example: a reviewer proposed a type-checking macro for a node's input when ConfigBag already did all that validation, so the clean fix was to route through ConfigBag, not to add parallel machinery. Worse, that kind of dirty fix often lands user-facing: in this project the people who author nodes are the same surface that gets reused as the language, so node-authoring must stay as minimal as possible, the node author writes their domain logic and the ctx (ExecutionContext) hands them the capabilities. A fix that makes the node author hand-roll plumbing the language should have provided is a bad fix even when it works. So before you accept any proposed fix, ask: does something already exist in the architecture that should be connected here instead, and would connecting it solve this more cleanly (DRY, keeps the node surface minimal, plugs into the generic mechanism rather than forking a side path). If yes, that connection IS the fix, override the agent's proposal. Only when nothing suitable exists do you accept a genuinely new fix, and even then reshape it clean rather than patch it. You hold the whole architecture and the full rules; the agent saw a slice; this is exactly where your override earns its keep. One specific caution here, and this should shape only your own judgment of their output, not the agents themselves: we may be deep into the round count on this feature, and something an agent calls wrong might actually be there for a reason we settled rounds ago. The failure mode I want to avoid: round 10 we chose arch A, round 15 we switched to arch B, round 17 we drift back toward A without realizing it was the original. So when a finding wants to change the shape (not just fix a clear bug), pause and ask whether that shape was a past deliberate decision before you walk it back. Don't ping-pong the architecture, drop key comments explaining why a shape is the way it is, and use the existing comments to tell whether we're ping-ponging.

Once you've got the validated findings, summarize for me. For the non-trivial-to-fix bugs, give me a bit of context (how it works now, where it would break) and how you'd fix it or a few ideas, technical but clear, not fuzzy, and don't assume I know every in-and-out of the architecture. For the trivial bugs, just a brief line each so I can skim for anything that smells wrong. The summary is informational, not a permission gate: do NOT ask "want me to proceed, want your call" when each bug has one obvious fix. Explaining how you'd fix something is not asking permission. Default to fixing everything in the same turn right after the summary. The only time you stop and ask is a genuine fork, two or more defensible fixes you can't decide between from my rules plus context (a real architectural tradeoff). "Non-trivial to explain" does not mean "needs my input," a bug can take a paragraph to describe and still have one obvious fix.

Then fix everything, and two things matter equally: the fixes have to be clean reshapes, and they must not introduce new bugs. Clean, not patched: I don't want patches, I want the base shape reshaped so the issue is impossible. Sometimes the right fix really is wiring a connection or adding a guard, but reduce added hops as much as you can. Stay DRY, if something will repeat then lift a parent, and if there are twenty near-identical parents, unify them. No spaghetti. A 1000-line reshape that gives the right shape beats a 10-line dirty patch we'll regret. And no new bugs: be extremely careful with each fix, take as long as you need, and use review agents to check your own fixes, after a non-trivial fix have an agent verify it does what you intend and introduces nothing new, iterate with them until each fix is genuinely clean.

It does not matter how non-trivial or time-consuming the proper fix is. No shortcuts. The goal is to let nothing ugly through to production. This round ends when you've fixed everything real you found, don't declare the feature clean or done, there'll be more rounds. Just report this round's findings and fixes and we keep going. Now go.
