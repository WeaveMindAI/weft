# Project Memory: Weft / Weavemind
Project-specific rules and facts for this working directory. General rules (em dashes, git push protocol, no fallbacks, decision framework, etc.) live in `~/.claude/CLAUDE.md`, not here.
**How to add memories:** add new entries inline here as `##` sections. Do NOT create separate `feedback_*.md` / `project_*.md` files in `memory/`; they don't auto-load and are invisible. Hard cap: 200 lines / 25 KB; content past that is silently dropped. Keep entries compressed like the existing ones (no `---`, no blank lines).
## Where design docs go (NOT weft/docs)
Temporary/design docs (redesign plans, fix-tracking, migration notes, anything that goes stale) live in `docs/` at the folder ROOT, never in `weft/docs/`. `weft/docs/` is ONLY user-facing always-true docs (authoring-nodes, getting-started, target-use-cases, loops); it ships with the OSS weft repo, so design notes there pollute it and rot. Never create `*-design.md` / `*-redesign.md` / `graph-edit-*.md` etc in `weft/docs/`. Past incident: created `docs/graph-edit-redesign.md` + `docs/graph-edit-fixes.md` in weft; Quentin removed them and said design docs belong in root `docs/` (the closed working area).
**Plan lifecycle:** an ACTIVE plan (working it now) lives in `docs/`. On a PARK signal ("keep it for later" / "park this"), MOVE it to `todo_plans/` at the repo root + add a short pointer entry in root `TODO.md` (plan holds all detail; TODO entry is just pointer + one-paragraph what/why). `/flow-2-plan` writes to `.claude/plans/` initially; parking is the move to `todo_plans/`.
## Weft (open) vs Weavemind (closed) repo split
Two folder-level boundaries in the monorepo:
- `/weft` = **open source**: the language + runtime (compiler, engine, dispatcher, listener, catalog, CLI, VS Code extension). Self-hosters run just this.
- `/weavemind` = **closed source**: hosted cloud (cloud-api, marketing site, k8s manifests, billing, auth glue).
**Why:** keep the language/runtime auditable by opening it; keep business logic (auth, payments, ops) closed.
**How to apply:** never import weavemind from weft (breaks self-hosters); the reverse is fine. When something belongs to both (e.g. start-execution billing), put the generic mechanism in weft and add cloud-only behavior (credits, gating) in weavemind; local/OSS mode no-ops or logs instead of enforcing. Check the path prefix before editing billing/auth/infra code.

## Loops: the iteration construct
`Loop` is a built-in keyword like `Group`: `name = Loop(in_ports) -> (out_ports) { config + decls + connections }`. Knobs: `over: [...]` (input ports whose `List[T]` outer is iterated as `T` inside), `carry: [...]` (output ports threaded across iterations, same name both sides). `parallel: Boolean` picks drive mode, default `false` (sequential) when absent; only explicit `true` switches, a non-boolean is a compile error (`loop-parallel-not-boolean`), never coerced (flatten materializes the default into LoopIn config so runtime always reads an explicit value). Optional: `max_iters` (absent = no cap), `trim_on_mismatch: Boolean` (default true; zip-trim vs panic on length mismatch). `over`/`carry` default empty.
**Four port roles:** iter input (`List[T]` out, `T` in), broadcast input (`T` both sides), gather output (out MUST be `List[T | Null]` explicit, in is `T?`), carry (same name both sides; out `T` in + `T` out; inside read `T` write `T?`). Implicit body ports: `self.index: Number` (read), `self.done: Boolean?` (write, sequential only). `index`/`done` are reserved names.
**Loop is a wrapper, not a lifecycle owner.** Outward emit happens once every launched iteration fired LoopOut AND a termination condition met; body work at past iterations' frame stacks (sub-agents, buses, branches not wired to LoopOut) keeps running afterward, addressed by frame stack so iterations can't collide. Enables the agent-swarm pattern (spawn N agents, expose `List[Bus | Null]` outward, agents stay alive talking to outside coordinators).
**Runtime shape (hidden from authors):** compiler lowers `Loop` to internal boundary node types `LoopIn`/`LoopOut` (NOT in catalog) keyed by the same `parent_group_id`. Engine's `LoopRuntime` holds one `LoopInstance` per `(loop_group_id, parent_frames, color)`, so nested/parallel-sibling/re-entrant runs each get an isolated instance. Dispatch in `execution_driver.rs::drive` intercepts LoopIn/LoopOut firings (and `Passthrough` group boundaries) BEFORE catalog lookup, runs them inline (no tokio task). LoopOut per-port closure: gather closed -> `null` at index; carry closed -> keep previous value; `done` closed -> `false`.
**Pause/resume:** four journal events (`LoopInstantiated`, `LoopIterationLaunched`, `LoopOutFired`, `LoopTerminated`) rebuild `LoopInstance` on fold via `rehydrate_loop_runtime`. Body nodes resume via the existing `await_signal` machinery (frame stack is already part of the per-(node, frames) key).
**Pulse-keying primitive `LoopFrames`** = stack of `LoopIteration { index: u32 }`, one frame per nested loop a firing is inside. Only loop machinery creates/reads it. Pulses match by EXACT frame stack.
**How to apply:** a node consuming a list carries the list verbatim (no implicit per-element fan-out); to process elements one by one, wrap the call site in `Loop(over: [...])`. Infra touching frame stacks uses the field `frames: LoopFrames`. Count-driven loop = `Range` node feeding `Loop(over: ["values"])`. User doc: `docs/loops.md`.
[Update Notice] If we redesign loop semantics (cascade inner loops on outer terminate, change carry type rules, new drive mode, move LoopIn/LoopOut out of inline dispatch into a spawned task), revisit.
## v2 architecture rules
**UX goal.** From a clean machine (weft CLI + Docker): dispatcher self-provisions, sidecars auto-provision/terminate per project lifecycle, trigger listeners auto-create/kill, liveness behind the scenes. State on disk + k8s; CLI is the sanctioned interface.
**Self-contained projects + nodes.** Project state (build outputs, image tags, infra, triggers, journal) scoped by project id so `weft clean <project>` wipes it. Nodes declare their full dependency set (cargo deps, system packages per manager/distro, env, ports, metadata, config) in their own dir. Multi-node packages share via `package.toml` unioned with the node's `deps.toml`.
**Local/cloud parity.** One code path: kind locally, real k8s remotely, same manifests. Dispatcher, listener, per-project worker images are the unit of deployment in both. Weft IS infrastructure; k8s is the native execution context.
**Surface-level review.** When a decision has architectural impact, pause and pull Quentin in before committing.
[Update Notice] If the build pipeline, runtime tier shape, or node packaging model changes, revisit.
## Weft as a real programming language
Weft is meant to become a real programming language, not just a workflow DSL. Properties to preserve: recursively compressible (groups nest arbitrarily), compilable (goal: a standalone binary where local nodes call in-process, no HTTP), typed (user-defined structs flow between nodes, not just JSON dicts), callbacks (nodes call back into the graph via colored pulses with callback metadata).
**Why:** product goal is "anyone can build anything, way faster." Current narrow scope is fine but every decision must preserve the path to the full language.
**How to apply:** ask "does this create a dirty patch that blocks compilation, richer types, or callbacks?" Never assume HTTP-only execution or untyped-JSON-only data flow.
[Update Notice] If we decide Weft stays a DSL and drops compilation/richer-types/callbacks, retire.
## Color = one execution (do not conflate)
`Color` (`weft_core::Color = Uuid`) IS the identity of a single execution: minted at `ExecutionStarted`, stamped on every event/pulse/firing, spent forever on terminate. A re-run is a NEW color. "Per color" ALWAYS means "per execution", never per-project or anything reused across runs. At most one worker drives a color at a time (journal stays one coherent stream). No "retry across colors", no per-color attempt cap (meaningless: done exec = gone color). A give-up-after-N-failures concept would be per-PROJECT (e.g. auto-deactivate a flapping project), explicitly out of scope. Past confusion: repeatedly proposed "per-color max attempts" which is meaningless.
[Update Notice] If Color's meaning changes (e.g. reused across re-runs), revisit.
## Runtime tiers: Listener / Dispatcher / Worker
Three tiers, one responsibility each.
**Listener.** Relays wake events (timer/webhook/form/SSE) to dispatcher; never decides/spawns/touches journal. POOLED + TENANT-AGNOSTIC (a pool of trusted control-plane pods, NOT one-per-tenant): placement is per-signal (`signal.listener_pod` + a `listener_pod` registry), load-based autoscale on memory pressure, drain-on-scaledown (re-places signals onto survivors then reaps). The old `tenant_listener` table + state machine are DELETED.
**Supervisor (infra).** Also POOLED + tenant-agnostic, mirroring the listener. Exclusive `infra_owner` lease per project (1:1 with its namespace) so exactly one supervisor runs kubectl per project; claims/renews via broker `sync_ownership`, work loops act only on owned projects; drains (releases leases for survivors) before reaping. Old scale-to-zero reaper + sentinel/heartbeat machinery removed.
**Dispatcher.** Pure routing + lifecycle + placement. Places pooled listener/supervisor pods; spawns worker Pods (one per PROJECT pool). Persists to Postgres (journal, signal table, leases, `listener_pod`/`infra_owner` registries). Does NOT execute node logic.
**Worker.** Runs the compiled project binary; one Pod per PROJECT pool, multiplexes N concurrent executions, idle-shuts-down. (Worker horizontal autoscale = capacity-overflow spawn = NOT YET BUILT, planned next; also planned: no-infra projects run namespace-LESS in a shared pool, only infra projects get a per-project namespace.) On stall + later fire, dispatcher spawns a fresh worker re-seeded from journal.
**Trust seam.** Broker `CallerScope::Tenant | ControlPlane`: pooled listener/supervisor are ControlPlane callers allowed to act for ANY tenant, validated per-op against the resource's real tenant. `mem_pressure` (cgroup reader + pure saturation/scaledown math in `weft-platform-traits`) is the SHARED autoscale primitive for both pools; local has no cgroup limit so pressure reads 0.0 (one pod, no special-case). One code path local and cloud. The infra raw-manifest extras hatch is GONE (weft owns the typed infra vocabulary, namespaced by construction).
**Why:** independent scale + failure. Dispatcher is the coordination plane (HA via Postgres leases); workers CPU-bound + ephemeral.
**Architecture 4 (implemented):** listener IS the kind-aware processor, dispatcher is pure transport.
- Listener owns `/register`, `/unregister`, `/process`. Registry stores kind spec + generic signal props (`is_resume`, `color`) needed at fire time.
- Listener's `process()` returns a kind-agnostic `ProcessAction` (ResolveSuspension / FireEntryTrigger / Hold / NoOp); dispatcher acts without inspecting kind.
- Dispatcher hosts `POST /signal/{token}`; webhooks come through dispatcher first (uniform parking gate).
- New kinds = new code in `crates/weft-listener/src/kinds/` only.
- Listener has no Postgres write authority; stateful kinds (Timer/SSE) call back to dispatcher's `/signal/internal-resume`.
**Bug to watch:** if listener can't return the right `ProcessAction`, do NOT add a dispatcher-side kind-specific override; plumb what the listener needs into `RegisterRequest` instead.
[Update Notice] If tier responsibilities change (workers persist, dispatcher executes nodes, listeners spawn workers), revisit.
## Dispatcher coordination through the DB
Dispatcher Pods are stateless coordination-wise: Postgres is the single source of truth, Pod-local RAM is just optimization.
**Ownership = leases.** Ownership is via DB leases (not exclusive RAM ownership); any Pod takes over by claiming an expired lease. Listener placement is per-signal (no per-tenant lease; `tenant_listener` is gone). Supervisor owns a project's infra via an exclusive `infra_owner` lease. Worker slots leased per project pool.
**Reconnect.** WS / in-RAM channels are fine, but reconnect to a sibling Pod must be cheap: no state preserved across the gap, the new Pod rebuilds from DB. If state can't be reconstructed from DB, it shouldn't exist.
**No implicit coordination.** Reject anything needing Pods to know each other's RAM, or assuming "same Pod handles next request" (Service load-balances). Reject best-effort race-window code that only works single-Pod.
**Rule for new features.** Before adding an in-RAM mutex/dashmap/counter on `DispatcherState`, ask "what if a sibling Pod handles the next request?" If the answer is stale reads or lost updates, it belongs in Postgres (advisory locks, row-level conflict, transaction).
**Why:** N dispatcher replicas is the production target; single-Pod is the degenerate case.
[Update Notice] If we change the coordination substrate (Redis for ephemeral, different slots/listeners store, sticky-session WS), revisit.
## weft-broker RBAC posture
`weft-broker` (in `weft-db` namespace) needs cluster-wide `create tokenreviews` permission. It authenticates worker/listener/sidecar pods via projected ServiceAccount tokens, validating them by calling kube-apiserver's `TokenReview` API (`authentication.k8s.io/v1/tokenreviews` create). TokenReview cryptographically verifies the token was minted by the cluster for the claimed SA + audience. Tenant pods live in per-project namespaces (`wm-project-<tenant>--<project>`); pooled control-plane pods (listener/supervisor) live in the control-plane namespace. The permission must be cluster-scoped: `weft-broker-sa` has a ClusterRoleBinding to a ClusterRole with `[{apiGroups:["authentication.k8s.io"], resources:["tokenreviews"], verbs:["create"]}]`. Don't narrow it by namespace; TokenReview is cluster-level. Namespace->tenant resolution uses the `weft_namespace_tenant` registry table (authoritative), NOT namespace-string parsing. The broker is otherwise unprivileged (no pod/service/deployment access, no namespace creation).
**Compromise model:** if pwned, the attacker can validate tokens (can't forge them; only kube-apiserver mints signed tokens), no escalation. The Postgres creds in broker RAM ARE valuable, but that's the accepted trust boundary (the broker fronts Postgres for user pods, full DB access).
## Language vs node separation
Two symmetric rules; check every design against both.
**Rule 1: the language is generic.** Compiler, dispatcher, engine, shared infra must never contain node-specific knowledge: no hardcoded field names, no node-type branches, no "if node is X" anywhere. The language exposes general features (wake signals, port resolution, form schemas, trigger subgraphs, sidecar provisioning); nodes are users of them.
**Rule 2: nodes do no plumbing.** Node code only does the node's own logic. Translating it into language concepts (registering a trigger, minting a URL, persisting a suspension, declaring a sidecar manifest) is the language's job; a node's `mod.rs` never reaches into the dispatcher directly.
When a node needs something the language lacks: design the general mechanism first, implement it as a language feature, then have the node use it. Never add a node-specific exception or hack.
**Why:** these two rules keep nodes self-contained and the language tiny; a violation either way creates coupling that ossifies the codebase. A node author thinks about their domain (Slack, LLMs, forms), not the runtime.
**How to apply:** ask "is node-specific knowledge leaking into the language? is plumbing leaking into a node?" If a node declares "I want behavior X", the language should expose a generic X any node could use. On a violation, redesign the general mechanism so neither side knows about the other; don't patch locally.
[Update Notice] If we redesign how nodes interface with the language (wake signal contract, sidecar protocol, ExecutionContext surface, codegen registry), revisit.
## No essays inside node implementations
Don't put paragraph-length comments next to catalog entries (port specs, field defs, helpers like markers/types). One line is fine; a paragraph isn't. Node files are touched often and skimmed fast, so paragraph comments at each call site are noise that ages badly. The home for "what does this marker mean" is the marker's definition in the shared module, written once. If you're writing more than one comment line next to `addsInputs: [...]` or a port spec, move the prose to the helper's definition.

## Append new k8s env vars at the end
When adding to a k8s `env:` list (or similar env block), append at the end; never insert mid-list or sort. Some code paths read env entries by index, not name, so reordering shifts indices and silently breaks unrelated lookups.

## Use setup.sh for rebuilds
For any rebuild or install in the weft repo, run `./setup.sh` at the repo root with flags for the components you want (`./setup.sh --help` for the set). It sequences the manual steps (cargo build, daemon refresh with image rebuild + pod rollout, pnpm bundle, vsix install, browser-extension build with version bump + optional AMO signing); doing them by hand silently leaves the dispatcher on stale code, the CLI on an old binary, or partial browser-extension artifacts.
**How to apply:** default to `./setup.sh`. Never invoke `pnpm build` / `pnpm zip` / `wxt` directly inside `extension-browser/`, even for a smoke test; use `./setup.sh --browser` (e.g. `--no-bump --chrome` for fast iteration). Missing a flag combo -> fix the script, not hand-run commands. To validate config WITHOUT building, use `pnpm exec tsc --noEmit` or `pnpm exec svelte-check`.

## No migration cruft
Pre-prod, fresh DB on every rebuild. Don't write `ALTER TABLE ADD COLUMN IF NOT EXISTS`, `DROP COLUMN IF EXISTS`, `RENAME COLUMN`, `INSERT ... SELECT` backfills, or `DROP TABLE IF EXISTS legacy`. Edit the canonical `CREATE TABLE` in place. Migration syntax pretends old data exists; on a fresh DB it's dead code that fragments the schema across many ALTERs. Past incident: a leftover `DROP TABLE IF EXISTS exec_event CASCADE` (from a one-time pre-v3 reset) wiped the journal on every dispatcher restart and stale-cursored the bridge into permanent silence.
**How to apply:** add a column -> edit the `CREATE TABLE` (NOT NULL DEFAULT or nullable). Remove a column/table/index -> delete it from the `CREATE`, no `DROP ... IF EXISTS`. Rename -> change the `CREATE`, no `RENAME COLUMN`. Backfill for new computed columns -> never; the writer fills it on the next run (cross-table denormalization like `execution_color` seeded from `ExecutionStarted` is a runtime concern, not migrate-time).
[Update Notice] If the project ever ships with production DB state worth preserving, retire this and add a real migration framework (sqlx-migrate, refinery) first.
## Layer-3 testing shape (subsystem rigs)
Shape for every new testable subsystem (supervisor done; listener, dispatcher, worker pending).
**Trait + fake colocation.** Each I/O dep gets a consumer-side trait in the consuming crate (e.g. `BrokerSupervisorOps` in `weft-infra-supervisor/src/broker_ops.rs`), whose file holds: trait def, `impl Trait for RealClient`, and `FakeX` (gated `#[cfg(any(test, feature = "test-helpers"))]`). Same pattern as `MockProjectStore`.
**Cross-cutting deps in `weft-platform-traits`.** k8s + clock are shared by supervisor/listener/(eventually)dispatcher. Trait + prod impl always shipped; fake gated behind `test-helpers`. Subsystems depend on `weft-platform-traits = { workspace = true }` for prod, add `features=["test-helpers"]` in `[dev-dependencies]`.
**State uses trait objects.** `SupervisorState` holds `Arc<dyn BrokerSupervisorOps>`, `Arc<dyn KubeClient>`, `Arc<dyn Clock>`. Never `Instant::now()` / `tokio::time::sleep` directly in subsystem code; always `state.clock.now()` / `.sleep()`.
**Loop body is `pub`, `run_loop` wraps.** `tick()` is public so tests step one iteration; `run_loop` is `loop { tick().await; clock.sleep(poll_interval).await }`.
**Rig at `src/testing/rig.rs`.** Public fields for fake access (`rig.broker`, `rig.kube`, `rig.clock`); `tick_*()` per loop; `advance(Duration)` for clock. No tokio spawning, no real sleep. Construct `SupervisorTestRig::with_tenant("...")`.
**Fakes are dumb.** `Mutex<Inner>` with state HashMaps + append-only `Vec<Call>` log; reads pull from maps, writes update maps AND push to log. Assert via helpers (`broker.events()`, `kube.scale_calls()`) or `calls()`. No business logic in fakes, no `mockall`/mock-DSL.
**Tests at `tests/*.rs`,** one file per loop (`tests/health_loop.rs`, etc). `use weft_x::broker_ops::{BrokerCall, BrokerSupervisorOps}` to call trait methods on `Arc<FakeBroker>`.
[Update Notice] If we change the rig shape (centralized test-helpers crate, generics instead of trait objects, a mock library), update.
## Don't timeout user-facing long operations
Weft is a programming language; real workflows (LLM pipelines, infra builds, human-in-the-loop pauses, executions) can legitimately run hours/days. Don't bind them with deadlines (a `--timeout 5m` on `weft run`, a 300s cap on `wait_for_drain`): that refuses the user's own program. For any wait on a user-controlled operation (build, deploy, infra, execution, drain, the user's node code, anything where a longer runtime could be legitimate), no deadline; make stuck-state legible instead: periodic breadcrumb (stderr in TTY, structured event in JSON) showing elapsed + what we wait on, plus a documented recovery (Ctrl+C, `weft stop <color>`, named verb). Deadlines belong only on internal service-to-service waits the user can't control (dispatcher->broker, supervisor->its own k8s call), where a hang is a bug. Unsure -> default to no-deadline + breadcrumb.
[Update Notice] If we add a new long-running CLI verb or change the drain/wait infra, revisit.
